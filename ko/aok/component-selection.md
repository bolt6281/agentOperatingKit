# 구성요소 선택 가이드

AOK는 정해진 단계를 따라 설치하는 구조도, 더 큰 프롬프트를 만드는 도구도 아니다. 프로젝트 상태와 사용자의 목적에 맞춰 필요한 구성요소만 선택해서 적용한다.

기본 경로와 파일 역할은 [권장 파일 구조](file-structure.md)를 따른다. 이 문서는 어떤 구성요소를 언제 선택할지 결정하는 기준만 둔다.

## 선택 원칙

- 루트 `AGENTS.md`는 매 세션 읽어도 부담 없는 크기로 유지한다. 작은 프로젝트는 50-100줄 안팎을 우선한다.
- 선택적 절차, 과거 기록, 문서 거버넌스 세부사항은 명시적 읽기 조건이 있는 문서로 분리한다.
- 처음부터 전체 AOK 구조를 만들지 않는다.
- 기존 문서 수정으로 충분하면 새 문서를 만들지 않는다.
- 사용자가 선택하지 않은 구성요소는 만들지 않는다.
- 생성, 수정, 이동, archive 대상 파일을 먼저 제안한다.
- 기존 문서의 의미를 바꾸는 변경은 사용자 승인 후 진행한다.
- 문서가 오래됐거나 충돌하면 source of truth를 즉시 확정하지 말고 충돌 후보로 남긴다.

AOK 적용의 성공 기준은 일반 작업에서 agent가 읽어야 하는 기본 컨텍스트가 줄어드는 것이다. AOK를 적용한 결과 모든 작업에서 더 많은 문서를 기본으로 읽게 된다면 적용이 잘못된 것이다.

## 읽는 빈도 분류

| 분류 | 파일 | 기본 동작 |
|---|---|---|
| 항상 읽음 | 루트 `AGENTS.md` | 매 세션 읽되 짧게 유지 |
| 라우팅 시 읽음 | `.agents/INDEX.md`, 하위 `AGENTS.md` | 라우팅이나 영역별 규칙이 필요할 때만 읽음 |
| 작업별로 읽음 | 선택된 workflow 파일 | 해당 반복 작업에만 읽음 |
| 참조가 필요할 때 읽음 | ADR, current status, version history | 작업이 해당 기준 문서에 의존할 때만 읽음 |
| 예외 상황에 읽음 | work log, archive | 롤백, 복구, 장애 조사, 회귀 추적, 과거 보존 확인 때만 읽음 |

## 구성요소별 선택 기준

| 구성요소 | 만들거나 갱신할 때 | 만들지 않을 때 |
|---|---|---|
| 루트 `AGENTS.md` | agent가 항상 알아야 하는 프로젝트 개요, 공통 명령어, 검증 라우팅, 짧은 문서 라우팅이 필요할 때 | README만으로 충분하고 agent 운영 규칙을 둘 필요가 없을 때 |
| 문서 지도 섹션 | 현재 프로젝트 문서의 읽는 조건과 source-of-truth 상태를 관리해야 할 때 | 문서가 한두 개뿐이고 README 라우팅으로 충분할 때 |
| agent 지침 거버넌스 섹션 | agent 지침 파일을 관리 대상 정책 파일로 두고 변경 승인 규칙이 필요할 때 | 작은 루트 `AGENTS.md` 하나만 있고 반복적인 지침 변경이 없을 때 |
| `.agents/INDEX.md` | workflow, 하위 agent 지침, 프로젝트 전용 skill을 색인해야 할 때 | 루트 `AGENTS.md` 하나로 충분하거나 Superpowers workflow를 쓰지 않을 때 |
| `.agents/workflows/*.md` | Superpowers를 사용하고 같은 작업 절차가 반복될 때 | Superpowers를 쓰지 않거나 반복 절차가 아직 안정되지 않았을 때 |
| `.agents/skills/<skill-name>/SKILL.md` | 같은 실패가 반복되고 테스트, lint, CI, 스크립트만으로 막기 어려운 판단 절차가 생겼을 때 | 자동화로 막을 수 있거나 절차가 아직 안정되지 않았을 때 |
| 하위 `AGENTS.md` | 특정 디렉터리나 영역에만 반복 적용되는 규칙, 검증 명령, 위험 경계가 있을 때 | repo 전체 규칙으로 충분하거나 적용 범위가 불명확할 때 |
| `docs/adr.md` 또는 `docs/adr/` | 제품, 아키텍처, UX, 데이터, 운영 기준의 결정 근거를 source of truth로 남겨야 할 때 | 단순 작업 기록이거나 기존 decision 문서로 충분할 때 |
| `docs/current-status.md` | 현재 운영 상태, 검증 기준, 배포/외부 의존성, 남은 위험을 짧게 유지해야 할 때 | README나 기존 운영 문서가 현재 기준을 충분히 담고 있을 때 |
| `docs/VERSION_HISTORY.md` | 버전, 릴리스 기준, 배포 산출물을 한 곳에서 추적해야 할 때 | 기존 CHANGELOG, release note, GitHub Releases, CONTRIBUTING으로 충분할 때 |
| `docs/work-log.md` | issue, PR, plan으로 추적하기 어려운 장기 운영 메모가 반복될 때 | 평소 작업 기록만 남기려는 목적이거나 기존 issue/PR/deploy log로 충분할 때 |
| `docs/archive/` | 현재 기준이 아닌 문서를 보존해야 할 때 | 삭제, 이동, deprecated 처리를 사용자에게 승인받지 않았을 때 |
| 전역 `~/.codex/AGENTS.md` | 사용자가 모든 프로젝트에 공통 적용할 개인 지침까지 AOK와 맞추고 싶을 때 | repo 내부 적용만 원하는 경우 |

## Workflow 선택 기준

Workflow 문서는 Superpowers를 사용하는 프로젝트에서만 선택한다. 스킬 호출을 일반 절차로 변환한 대체 workflow는 만들지 않는다.

| workflow | 선택할 때 | 보류할 때 |
|---|---|---|
| `feature-change.md` | 기능 추가나 동작 변경이 반복되고 요구사항, 범위, 검증 절차가 자주 흔들릴 때 | 단발성 작은 수정만 있는 경우 |
| `bugfix.md` | 재현, 원인 분석, 회귀 확인 절차가 반복될 때 | 단순 오탈자나 명확한 작은 수정일 때 |
| `refactor.md` | 외부 동작 보존과 내부 구조 개선을 분리해서 관리해야 할 때 | 기능 변경과 구조 변경이 아직 구분되지 않은 초기 작업일 때 |
| `docs-change.md` | 문서, 정책, ADR, agent 지침 변경이 반복될 때 | AOK를 처음 적용하는 중일 때. 이 경우 `application-guide.md`를 우선한다. |
| `risky-change.md` | DB, migration, 배포, CI, secret, 인증/권한, 결제, 보안처럼 실패 비용이 큰 변경이 있을 때 | 위험 변경이 거의 없거나 기존 승인 절차가 충분할 때 |
| `research-decision.md` | 조사, 비교, 기술 선택, 의사결정 근거 정리가 반복될 때 | 단순 검색이나 일회성 참고 자료일 때 |
| `review-feedback.md` | 리뷰 피드백의 타당성 확인과 범위 조정이 반복될 때 | 리뷰 피드백이 간단하고 바로 반영 가능한 경우 |
| `repeated-failure.md` | 같은 실패가 반복되어 자동화, 문서, skill 중 무엇으로 막을지 판단해야 할 때 | 원인이 일회성이거나 아직 반복 패턴이 확인되지 않았을 때 |

## 기존 문서가 있을 때

- 이미 있는 문서는 삭제하거나 이동하지 않는다.
- 새 AOK 문서를 자동으로 추가하지 않는다.
- 문서 지도를 선택한 경우 기존 문서의 상태와 기준 여부를 기록한다. 선택하지 않은 경우 루트 `AGENTS.md`에는 최소 문서 라우팅만 둔다.
- 기존 `AGENTS.md`가 크면 루트 라우터, `.agents/INDEX.md`, workflow, 일반 docs로 분해할 후보를 먼저 제안한다.
- 기존 CHANGELOG나 release note가 있으면 `docs/VERSION_HISTORY.md`보다 기존 문서 보강을 우선한다.
- 기존 ADR, current status, archive가 있으면 새 구조로 갈아엎지 말고 현재 기준 문서인지, 보존 문서인지, 충돌 후보인지부터 표시한다.

## 추가 검토 신호

다음 패턴이 반복되면 관련 구성요소를 추가할지 검토한다.

- 같은 작업 절차를 매번 설명하고 있다.
- agent가 같은 유형의 실수를 반복한다.
- 문서 위치를 찾는 데 시간이 든다.
- 검증 명령이나 작업 경계가 자주 혼동된다.
- 의사결정 이력이 코드와 어긋나기 시작한다.
- 롤백, 복구, 장애 조사, 회귀 추적 때 과거 흐름을 찾기 어렵다.
