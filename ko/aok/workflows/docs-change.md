# 문서 / 정책 / ADR 변경 Workflow

> 관리 대상 agent workflow 문서입니다. 사용자 승인 없이 의미를 바꾸는 수정은 하지 않습니다.
> 전제: 이 workflow는 Superpowers 스킬이 설치된 agent 환경을 기준으로 한다.

## 목적

문서 난립을 막고, 기존 문서와 정책의 일관성을 유지한다.

## 절차

1. 새 문서를 만들기 전에 기존 문서로 해결 가능한지 먼저 확인한다.
2. 기존 문서 수정이나 섹션 추가로 충분하면 해당 문서를 우선 갱신한다.
3. 기존 프로젝트 문서를 AOK 구조에 반영할 때는 원본 보존, archive 보존, 병합 우선, 충돌 보고 중 사용자가 선택한 방식을 따른다.
4. 기존 문서끼리 내용이 충돌하면 임의로 하나를 기준으로 정하지 않고 충돌 목록과 추천 기준을 먼저 제안한다.
5. 새 문서가 필요하면 문서명, 경로, 목적, 기존 문서로 해결되지 않는 이유를 먼저 제안하고 사용자 승인 후 생성한다.
6. 제품 핵심 결정이 바뀌면 `docs/adr.md` 또는 `docs/adr/INDEX.md`가 가리키는 관련 영역 ADR을 우선 갱신한다.
7. 문서 변경이 구현 계획으로 이어지면 `brainstorming` 또는 `writing-plans`를 사용한다.
8. 완료 전 관련 문서 간 충돌 여부와 정보 소실 여부를 확인하고 `verification-before-completion`으로 검증한다.

## 문서 원칙

- 문서 작업의 기본 원칙은 기존 문서 갱신 우선이다.
- 문서 확인이나 문서 수정 필요가 곧 신규 문서 생성 허용을 뜻하지 않는다.
- 새 Markdown 문서가 필요하다고 판단되면, 문서명, 경로, 목적, 기존 문서로 해결되지 않는 이유를 먼저 제안하고 사용자 승인 후 생성한다.
- 제품 핵심 결정이 바뀌면 먼저 `docs/adr.md` 또는 관련 `docs/adr/<영역>/<결정-주제>.md`를 갱신한다.
- 프로젝트 전용 agent 문서도 새 Markdown 문서에 해당하므로, 새로 만들 때는 먼저 사용자 승인을 받는다.
- 기존 문서를 AOK 양식으로 병합할 때는 원본의 정보, 예외, 세부 조건이 빠지지 않았는지 확인한다.
- 기존 문서 이동, 삭제, deprecated, archived 처리는 사용자 승인 없이 하지 않는다.
- 기존 workflow와 AOK workflow가 충돌하면 사용자가 선택한 충돌 처리 방식에 따라 기존 규칙 우선, AOK 규칙 우선, 혼합 적용, 충돌 목록 승인 중 하나로 처리한다.

## 완료 전 확인

- 기존 문서 수정으로 해결 가능한지 확인했는가?
- 새 문서 생성이 필요하다면 사용자 승인을 받았는가?
- 관련 문서 간 충돌 여부를 확인했는가?
- 기존 문서의 정보 소실 가능성을 확인했는가?
- 기존 파일 이동, 삭제, deprecated, archived 처리는 승인받았는가?
