# AOK 적용 가이드라인

AOK는 agent 지침, 작업 workflow, 문서 지도, 문서 거버넌스를 정리해 프로젝트 진행 중 컨텍스트 낭비와 문서 불일치를 줄이는 운영 키트다.

AOK를 적용하라는 요청을 받은 agent는 바로 파일을 만들지 않는다. 먼저 AOK가 무엇인지 설명하고, 현재 프로젝트 상태를 확인한 뒤, 단계적으로 선택지를 제시한다.

## 적용 순서

1. AOK의 역할을 짧게 설명한다.
2. 기존 프로젝트라면 현재 파일 구조, 기존 agent 지침, README, docs, workflow, 테스트/빌드 명령을 확인한다.
3. 1차 적용 인터뷰를 선택지로 제시한다.
4. 사용자 선택에 따라 필요한 2차 질문만 제시한다.
5. 추천 채택 레벨과 적용안을 만든다.
6. 생성, 수정, 이동, archive, deprecated 처리할 파일 목록과 문서 흡수 추적표를 먼저 제안한다.
7. 사용자 승인 후에만 실제 파일을 생성하거나 기존 문서의 의미를 바꾸는 수정을 한다.

## 현재 프로젝트 진단

기존 프로젝트에 AOK를 적용할 때는 다음 항목을 먼저 확인한다.

- 기존 agent 지침: `AGENTS.md`, `agent.md`, `CLAUDE.md`, `GEMINI.md`, `.cursor/rules`, `.windsurfrules`
- 프로젝트 설명: `README.md`, `docs/`, planning 문서, architecture 문서
- 작업 절차: issue/PR 템플릿, workflow 문서, release 문서, 배포 문서
- 검증 방법: package scripts, Makefile, CI 설정, test/lint/typecheck/build 명령
- 문서 상태: 중복 문서, 오래된 문서, 충돌하는 기준, source of truth 후보
- Superpowers 사용 흔적: `docs/superpowers/`, `.superpowers/`, 기존 skill 이름, spec/plan 문서, Superpowers 기반 workflow 문구

확인 결과는 사용자에게 짧게 요약한다. 기존 문서에 충돌이 있으면 임의로 정리하지 말고 충돌 후보로 표시한다.

## 1차 적용 인터뷰

처음에는 아래 세 가지만 묻는다. 적용 목적은 복수 선택 가능하다.

### 1. AOK 적용 목적

- 신규 개발 준비: 기획만 있는 repo에서 본격 개발을 시작하기 위한 agent 운영 구조가 필요하다.
- 작업 파이프라인 정립: 기능 추가, 버그 수정, 리팩터링, 문서 변경 절차를 안정화하고 싶다.
- 문서 정합성 회복: 코드와 문서, 문서와 문서 사이의 괴리를 줄이고 source of truth를 정리하고 싶다.
- 컨텍스트 절감: 큰 문서를 기능별로 나누고 색인해 agent가 필요한 문서만 읽게 하고 싶다.
- 기존 프로젝트 정비: 이미 있는 README, agent 지침, docs, planning 문서를 AOK 구조에 맞게 흡수하고 싶다.
- 협업/장기 운영 강화: 여러 사람 또는 여러 agent가 작업해도 의사결정과 작업 이력이 유지되게 하고 싶다.
- 위험 작업 통제: DB, 배포, CI, 의존성, 인증/권한 같은 고위험 변경 절차를 엄격히 관리하고 싶다.

### 2. 프로젝트 상태

- 신규 repo: 기존 문서나 workflow가 거의 없다.
- 기존 repo: README, docs, agent 지침, workflow 중 일부가 이미 있다.
- 문서 혼선 repo: 기준 문서가 충돌하거나 오래된 문서가 많다.
- 진단만 먼저: 실제 변경 없이 구조와 적용안만 보고 싶다.

### 3. Superpowers 기반 workflow 사용 여부

- 사용함: Superpowers가 이미 설치되어 있거나 설치 후 AOK workflow를 적용한다.
- 설치 안내 필요: workflow를 쓰고 싶지만 Superpowers 설치 방법이 필요하다.
- 사용하지 않음: AOK workflow는 적용하지 않고 루트 `AGENTS.md` 중심으로만 적용한다.
- 나중에 결정: 지금은 workflow 생성을 보류한다.

Superpowers를 사용하지 않는 경우 `.agents/workflows/*.md`는 제공하지 않는다. 스킬 호출을 일반 절차로 변환한 workflow도 만들지 않는다.

Superpowers 사용 여부가 불명확하면 아래 기준으로 후보를 판단하고, 사용자에게 확인한다.

- 사용 가능 후보: `docs/superpowers/` 산출물, `.superpowers/` 설정, 기존 `AGENTS.md`의 `brainstorming`, `writing-plans`, `test-driven-development`, `verification-before-completion` 같은 skill 이름, 기존 spec/plan/review 문서가 있다.
- 확인 필요 후보: Superpowers 이름은 없지만 agent workflow 문서나 plan/spec 산출물이 반복적으로 있다.
- 미사용 후보: skill 호출 흔적이 없고, 사용자가 설치 또는 사용 의사가 없다고 답한다.

## 2차 적용 인터뷰

1차 선택 결과에 따라 필요한 질문만 추가한다.

### AOK 핵심 파일 적용 범위

- 루트 `AGENTS.md`: 프로젝트 개요, 공통 명령어, 검증 라우팅, 문서 지도를 담는다.
- 전역 `~/.codex/AGENTS.md`: 사용자가 모든 프로젝트에 공통 적용할 개인 원칙까지 AOK에 맞추고 싶을 때만 선택한다.
- `.agents/INDEX.md`: agent workflow와 지침 파일의 색인을 관리한다.
- `.agents/workflows/*.md`: Superpowers 사용 프로젝트에서만 작업별 절차를 분리한다.
- `docs/adr.md` 또는 `docs/adr/`: 제품, 아키텍처, UX, 데이터, 운영 기준의 source of truth를 둔다. ADR이 커지면 번호식 파일 나열보다 `docs/adr/<영역>/<결정-주제>.md` 구조를 우선한다.
- `docs/current-status.md`: 현재 운영 상태, 검증 기준, 배포/외부 의존성, 남은 위험을 짧게 기록한다.
- `docs/VERSION_HISTORY.md`: 주요 변경과 의사결정 이력을 버전 기준으로 기록한다.
- `docs/work-log.md`: issue, PR, plan으로 추적하기 어려운 장기 운영 메모가 있을 때만 만든다.
- 하위 `AGENTS.md`: 특정 영역에만 적용되는 규칙이 있을 때 별도 지침을 둔다.
- 지금은 선택하지 않음: 최소 구조만 적용하고 나중에 필요할 때 승격한다.

### 기존 문서 처리 방식

AOK는 기존 문서를 삭제하지 않고 먼저 읽은 뒤, 정보 손실 없이 새 구조에 반영하는 것을 원칙으로 한다.

- 원본 보존 우선: 기존 문서는 그대로 두고, AOK 문서에서 링크하거나 요약한다.
- archive 보존: 기존 문서를 `docs/archive/`로 이동하고, 현재 기준 문서에는 정리된 내용만 남긴다.
- 병합 우선: 기존 문서 내용을 AOK 양식에 최대한 흡수하고, 중복 문서는 deprecated 처리한다.
- 충돌 보고 우선: 기존 문서끼리 내용이 다르면 바로 정리하지 않고 충돌 목록을 사용자에게 먼저 보고한다.
- 변경 전 매번 승인: 기존 문서 이동, 삭제, deprecated 표시는 항상 사용자 승인 후 처리한다.

### 기존 작업 파이프라인과 AOK workflow 충돌 처리

- 기존 규칙 우선: 이미 프로젝트에 정의된 workflow를 기준으로 하고 AOK는 보조 규칙으로 반영한다.
- AOK 규칙 우선: 기존 workflow를 AOK workflow에 맞게 재구성한다.
- 혼합 적용: 기존 규칙 중 유효한 것은 유지하고, 빈 부분만 AOK로 채운다.
- 충돌 목록 승인: 충돌 항목을 표로 정리한 뒤 사용자 승인 전까지 수정하지 않는다.

### 기존 `AGENTS.md` 분해 방식

기존 `AGENTS.md`가 크거나 workflow를 직접 담고 있으면, 아래 기준으로 분해안을 제안한다.

| 보낼 위치 | 남길 내용 |
|---|---|
| 루트 `AGENTS.md` | 항상 읽혀야 하는 프로젝트 개요, 코드베이스 구조, 공통 명령어, 검증 라우팅, 핵심 금지/승인 규칙, 문서 지도 |
| `.agents/INDEX.md` | workflow 라우팅, 승인된 agent 지침 파일 목록, 프로젝트 전용 skill 색인, AOK 적용 결정 요약 |
| `.agents/workflows/*.md` | 기능 추가, 버그 수정, 리팩터링, 문서 변경, 위험 변경, 리뷰 피드백 같은 반복 절차 |
| 하위 `AGENTS.md` | 특정 디렉터리나 영역에서만 반복되는 코드 작성 규칙, 검증 명령, 디자인/운영 규칙 |
| 일반 docs | 제품 결정, 운영 상태, 버전 이력, 배포 절차, 장기 메모처럼 agent 지침보다 프로젝트 기준 문서에 가까운 정보 |

판단이 애매하면 루트 `AGENTS.md`에는 짧은 라우팅 규칙만 남기고, 상세 규칙은 후보 위치와 이유를 적용안에 적는다.

### 문서 관리 엄격도

- 최소 관리: 문서 지도 없이 루트 `AGENTS.md`에 핵심 문서 위치만 적는다.
- 느슨한 관리: agent가 필요한 문서를 자유롭게 갱신하되, 새 문서 생성은 제안 후 진행한다.
- 표준 관리: 기준 문서, archive, deprecated 상태를 문서 지도에 기록한다.
- 엄격 관리: source of truth, 문서 상태, 마지막 검토일, 충돌 처리 규칙을 강제한다.

### 기존 파일 변경 승인 방식

- 적용안 일괄 승인: 생성, 수정, 이동, archive 대상 목록을 한 번에 승인한 뒤 진행한다.
- 파일별 승인: 기존 파일을 수정하거나 이동할 때마다 사용자 승인을 받는다.
- 원본 백업 후 진행: 기존 파일을 바꾸기 전에 원본을 archive 또는 백업 경로에 보존한다.
- 읽기 전용 진단 먼저: 실제 변경 없이 진단 보고서와 적용안만 만든다.
- 전역 파일 제외: `~/.codex/AGENTS.md`는 확인하거나 수정하지 않고 repo 내부 파일만 대상으로 한다.

## 선택 결과 반영 규칙

| 사용자 선택 | AOK 반영 방식 |
|---|---|
| 신규 개발 준비 | 루트 `AGENTS.md`와 기본 명령어, 검증 라우팅을 우선 만든다. |
| 전역 `~/.codex/AGENTS.md` | 기존 전역 지침을 확인한 뒤 추가/병합/유지/건너뛰기 선택지를 제안하고, 사용자 승인 후에만 반영한다. |
| 작업 파이프라인 정립 | Superpowers를 사용하는 경우에만 `.agents/INDEX.md`와 필요한 `.agents/workflows/*.md`를 적용한다. |
| Superpowers 설치 안내 필요 | 현재 LLM 환경에 맞는 Superpowers 설치 방법을 먼저 안내하고, 설치 후 workflow 적용 여부를 다시 확인한다. |
| Superpowers 사용하지 않음 | workflow 문서를 만들지 않고 루트 `AGENTS.md`의 기본 작업 원칙만 적용한다. |
| 문서 정합성 회복 | 문서 지도, source of truth 표시, 충돌 목록, archive 정책을 우선 적용한다. |
| 컨텍스트 절감 | 긴 문서를 기능별로 분리하고, 필요한 문서만 읽는 색인 규칙을 강화한다. |
| 기존 프로젝트 정비 | 기존 문서를 하나씩 읽어 AOK 양식에 반영하고, 소실 위험이 있는 내용은 추적표에 표시한다. |
| 협업/장기 운영 강화 | ADR, current-status, VERSION_HISTORY, 필요한 하위 `AGENTS.md` 후보를 검토한다. 기존 Level 3 성격 문서가 이미 있으면 새로 만들지 말고 보존 또는 정리 후보로 표시한다. |
| 위험 작업 통제 | Superpowers 사용 시 risky-change workflow, 승인 필요 작업, 롤백/검증 규칙을 우선 적용한다. |
| 원본 보존 우선 | 기존 문서를 삭제하거나 이동하지 않고 링크와 요약 중심으로 연결한다. |
| archive 보존 | 현재 기준에서 제외되는 문서를 `docs/archive/` 후보로 제안한다. |
| 병합 우선 | 중복 내용을 기준 문서에 병합하되, 병합 전후 정보 소실 여부를 확인한다. |
| 충돌 목록 승인 | 충돌 표를 먼저 만들고 사용자 결정 전까지 기준 문서를 확정하지 않는다. |

## 기존 문서 흡수 추적표

기존 프로젝트 문서를 AOK 구조에 반영할 때는 적용안에 아래 표를 포함한다.

| 원본 경로 | 추출한 정보 | 반영 대상 | 보존 방식 | 충돌 여부 | 누락 위험 |
|---|---|---|---|---|---|
| `[기존 문서]` | `[핵심 정보]` | `[새 위치 또는 유지 위치]` | 유지 / 병합 / archive / deprecated 후보 | 있음 / 없음 | 낮음 / 중간 / 높음 |

누락 위험이 중간 이상인 항목은 사용자의 확인 전까지 삭제, 이동, deprecated 처리하지 않는다.

## 적용안 제안 형식

사용자 선택을 받은 뒤 agent는 아래 형식으로 적용안을 제안한다.

```md
## AOK 적용안

- 적용 목적:
- 추천 채택 레벨:
- 혼합 채택 여부:
- 생성할 파일:
- 수정할 파일:
- 보존 또는 이동할 기존 파일:
- deprecated 또는 archived 처리 후보:
- Superpowers 설치 또는 workflow 보류 여부:
- 기존 workflow와 AOK workflow 충돌:
- 문서 흡수 추적표:
- 정보 소실 방지 방식:
- 확인 필요 항목:
- 적용 전 사용자 승인 필요: 예
```

사용자가 승인하기 전에는 파일을 생성하거나 기존 문서의 의미를 바꾸지 않는다.

## 적용 결과물 품질 규칙

- 최종 파일에는 `[명령어]`, `[경로]`, `[문서명]`, TODO 같은 미해결 placeholder를 남기지 않는다.
- 확인하지 못한 값은 본문에 억지로 넣지 않고 적용안의 `확인 필요 항목`에 둔다.
- 예시가 필요한 경우에는 실제 프로젝트 값과 섞이지 않도록 `예시:`라고 표시한다.
- 새 Markdown 문서가 필요해 보여도 먼저 기존 문서 수정으로 해결 가능한지 확인한다.

## 기존 프로젝트 적용 원칙

- 기존 파일은 하나씩 읽고, AOK 양식에 녹일 수 있는 정보를 최대한 반영한다.
- 놓치거나 소실될 수 있는 정보는 삭제하지 않고 보존 후보, 충돌 후보, 확인 필요 항목으로 표시한다.
- 기존 작업 파이프라인과 AOK workflow가 겹치면 선택한 충돌 처리 방식에 따라 적용한다.
- 기존 문서의 원본 보존 여부를 확인하지 않은 상태에서 삭제, 이동, deprecated 처리를 하지 않는다.
- 현재 기준 문서가 무엇인지 불명확하면 임의로 source of truth를 정하지 말고 후보와 근거를 제안한다.
