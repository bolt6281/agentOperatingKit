# ADR 템플릿

이 템플릿은 제품, 아키텍처, UX, 데이터, 운영 기준처럼 이후 작업의 기준이 되는 결정을 기록할 때 사용한다.

작은 프로젝트에서는 `docs/adr.md` 안에 이 양식을 반복해서 사용할 수 있다. ADR이 많아지면 `docs/adr/INDEX.md`와 `docs/adr/<영역>/<결정-주제>.md` 구조로 분리한다.

## `docs/adr/INDEX.md` 예시

| 영역 | ADR 경로 | 상태 | 읽는 조건 |
|---|---|---|---|
| product | `docs/adr/product/notification-policy.md` | accepted | 알림 정책, 사용자 흐름, 기능 범위 변경 |
| architecture | `docs/adr/architecture/app-web-boundary.md` | accepted | 앱/웹/서버 경계 변경 |
| data | `docs/adr/data/job-status-model.md` | proposed | 상태 모델, DB schema, migration 변경 |

## ADR 본문 복사 범위

실제 ADR 파일을 만들 때는 아래 `ADR 본문 양식`부터 `적용 시 주의` 직전까지 복사한다. 이 템플릿 설명과 `docs/adr/INDEX.md` 예시는 실제 ADR 본문에 포함하지 않는다.

## ADR 본문 양식

# ADR: `[결정 주제]`

| 항목 | 값 |
|---|---|
| 상태 | proposed / accepted / deprecated / superseded |
| 결정일 | `[YYYY-MM-DD]` |
| 적용 범위 | `[제품 / 아키텍처 / 데이터 / 운영 / UX / 기타]` |
| 소유자 | `[사람 또는 팀]` |
| 관련 문서 | `[링크]` |
| 대체한 ADR | `[없음 또는 경로]` |

## 배경

- `[어떤 문제나 제약 때문에 결정이 필요한지]`
- `[현재 기준 문서, 코드 상태, 운영 조건]`

## 결정

`[최종 결정 내용을 짧고 명확하게 적는다.]`

## 이유

- `[이 결정을 선택한 이유]`
- `[중요한 trade-off]`

## 검토한 대안

| 대안 | 장점 | 단점 | 채택하지 않은 이유 |
|---|---|---|---|
| `[대안]` | `[장점]` | `[단점]` | `[이유]` |

## 영향

- 코드 영향: `[영향 범위]`
- 문서 영향: `[갱신해야 하는 문서]`
- 운영 영향: `[배포, 롤백, 모니터링, 데이터 migration 등]`
- 위험: `[남는 위험 또는 확인 필요 항목]`

## 검증 또는 운영 기준

- `[이 결정이 지켜지는지 확인하는 방법]`
- `[관련 테스트, 리뷰 기준, 배포 전 확인 항목]`

## 변경 이력

| 날짜 | 변경 | 이유 |
|---|---|---|
| `[YYYY-MM-DD]` | 최초 작성 | `[이유]` |

## 적용 시 주의

- 실제 프로젝트에 적용할 때는 placeholder를 남기지 않는다.
- `ADR 템플릿`, `docs/adr/INDEX.md 예시`, `ADR 본문 복사 범위` 섹션은 실제 ADR 파일에 복사하지 않는다.
- 아직 결정되지 않은 값은 본문에 억지로 채우지 말고 `확인 필요 항목`으로 분리한다.
- 상태가 `superseded`라면 대체 ADR 경로를 반드시 적는다.
