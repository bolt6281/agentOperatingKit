# 작업 로그 템플릿

이 템플릿은 issue, PR, plan, release note만으로 추적하기 어려운 장기 구현 흐름과 운영 메모를 기록할 때 사용한다.

`docs/work-log.md`는 현재 기준 문서가 아니다. 현재 운영 기준은 `docs/current-status.md`, 버전별 릴리스 기준은 `docs/VERSION_HISTORY.md`, 결정 근거는 ADR에 둔다.
평소 작업 시작 시에는 읽지 않고 기록 대상으로만 사용한다. 롤백, 복구, 장애 조사, 회귀 원인 추적, 오래된 구현 이유 확인처럼 과거 흐름이 필요한 경우에만 읽는다.

# 작업 로그

> 장기 구현 이력과 운영 메모.

## 작성 규칙

- 최신 항목을 위에 둔다.
- 한 항목은 날짜, 짧은 제목, 변경 요약, 검증으로 끝낸다.
- 장기적으로 참고할 구현 이유나 운영 판단만 남긴다.
- 평소 작업에서는 기록만 하고, 새 작업의 기준 문서로 읽지 않는다.
- 단순 진행 상황, 채팅 요약, 임시 조사 기록은 누적하지 않는다.
- secret, 개인정보, 전체 로그 덤프, 대용량 명령 출력은 남기지 않는다.

## 읽는 경우

- 롤백 또는 복구 작업에서 이전 변경 흐름을 확인할 때
- 장애, 회귀, 데이터 불일치의 원인을 과거 작업과 대조할 때
- 오래된 구현 이유나 운영 판단을 확인해야 할 때
- 문서 간 기준이 충돌해 변경 이력을 추적해야 할 때

### `[YYYY-MM-DD] [짧은 제목]`

- `[무엇을 바꿨는지]`
- `[왜 바꿨는지 또는 어떤 기준 문서/요청과 연결되는지]`
- `[영향을 받은 주요 영역 또는 산출물]`
- 검증:
  - `[검증 명령 또는 확인 방법]`

## 적용 시 주의

- 적용 완료된 파일에는 placeholder를 남기지 않는다.
- 같은 내용이 release baseline을 바꾸면 `docs/VERSION_HISTORY.md`도 함께 갱신한다.
- 제품, 아키텍처, 데이터, 운영 정책 결정이 바뀌면 work log가 아니라 ADR을 갱신한다.
