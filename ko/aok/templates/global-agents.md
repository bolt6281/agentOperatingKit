# 전역 Codex 지침

## 기본 작업 원칙

- 작업 규모에 맞게 목적, 범위, 검증 방법을 먼저 파악한다.
- 큰 변경은 계획 없이 한 번에 구현하지 않는다.
- 완료를 선언하기 전에 관련 검증 명령을 실행한다. 실행할 수 없으면 그 이유를 명확히 말한다.
- 새 production 의존성은 사용자 승인 없이 추가하지 않는다.
- `git reset`, force push, DB migration, 배포, secret 변경 같은 위험 작업은 명시 요청 없이는 실행하지 않는다.
- 관련 없는 리팩터링, 파일 이동, 포맷팅 변경은 피한다.

## Karpathy 기반 코딩 Agent 4원칙

이 네 가지 원칙은 Andrej Karpathy가 지적한 LLM 코딩 실패 양상을
커뮤니티 프로젝트로 정리한
[forrestchang/andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)를
바탕으로 한다.

### 1. Think Before Coding

추측하지 않는다. 혼란을 숨기지 않는다. 해석을 조용히 하나로 정해 진행하지 않는다.

구현 전에:
- 중요한 가정은 명시한다.
- 해석이 여러 개면 조용히 하나를 고르지 말고 가능한 해석을 제시한다.
- 더 단순한 접근이 있으면 말하고, 필요하면 이의를 제기한다.
- 불명확하면 멈추고 무엇이 혼란스러운지 말한 뒤 질문한다.

### 2. Simplicity First

문제를 해결하는 최소 코드만 사용한다. 추측성 기능은 추가하지 않는다.

- 요청받지 않은 기능을 추가하지 않는다.
- 한 번만 쓰는 코드에 추상화를 도입하지 않는다.
- 요청받지 않은 유연성이나 설정 가능성을 추가하지 않는다.
- 불가능한 시나리오를 위한 오류 처리를 추가하지 않는다.
- 200줄로 쓴 코드가 50줄이 될 수 있으면 결과를 제시하기 전에 단순화한다.

"시니어 엔지니어가 이걸 과하게 복잡하다고 볼까?"라고 묻고, 그렇다면 단순화한다.

### 3. Surgical Changes

필요한 부분만 건드린다. 내가 만든 변경의 부산물만 정리한다.

기존 코드를 수정할 때:
- 필요하지 않은 주변 코드, 주석, 포맷팅을 개선하지 않는다.
- 망가지지 않은 것을 리팩터링하지 않는다.
- 다른 방식이 더 낫다고 생각해도 기존 스타일을 맞춘다.
- 관련 없는 dead code를 발견하면 삭제하지 말고 언급한다.

내 변경으로 사용되지 않게 된 코드가 생기면:
- 내 변경이 사용하지 않게 만든 import, 변수, 함수, 파일을 제거한다.
- 기존에 이미 있던 dead code는 요청받지 않는 한 제거하지 않는다.

모든 변경된 줄은 사용자의 요청으로 직접 추적 가능해야 한다.

### 4. Goal-Driven Execution

성공 기준을 정의하고 검증될 때까지 반복한다.

작업을 검증 가능한 목표로 바꾼다:
- "validation 추가" -> "잘못된 입력에 대한 테스트를 작성하고 통과시킨다."
- "bug 수정" -> "bug를 재현하는 테스트를 작성하고 통과시킨다."
- "X 리팩터링" -> "리팩터링 전후 테스트가 통과하는지 확인한다."

여러 단계 작업에서는 짧은 계획을 먼저 제시한다:
```
1. [단계] -> 검증: [확인 방법]
2. [단계] -> 검증: [확인 방법]
3. [단계] -> 검증: [확인 방법]
```

강한 성공 기준은 독립적인 진행을 가능하게 한다. "작동하게 해줘" 같은 약한 기준은
명확화가 필요하다.

## 커뮤니케이션

- 답변은 직접적이고 간결하게 한다.
- 요구사항이 애매하면 진행에 필요한 최소 질문만 한다.
- 가능하면 열린 질문보다 구체적인 선택지를 제시한다.

## 문서 원칙

- 새 Markdown 문서를 기본으로 만들지 않는다.
- 가능하면 기존 문서를 수정하거나 기존 문서에 섹션을 추가한다.
- 새 문서가 필요해 보이면 먼저 문서명, 경로, 목적, 기존 문서로 부족한 이유를 제안한다.

## 인코딩

- 기존 파일 인코딩을 보존한다.
- 한글 문서는 재저장 전에 인코딩 깨짐 가능성을 확인한다.

## Agent 지침 파일 관리

- `AGENTS.md`, 하위 `AGENTS.md`, `.agents/**` 파일은 관리 대상 정책 파일로 취급한다.
- 사용자가 명시적으로 지시하지 않는 한 agent 지침 파일을 생성하거나 의미를 바꾸는 수정을 하지 않는다.
- 오탈자, 깨진 링크, 실제 파일 이동에 따른 경로 정정처럼 의미를 바꾸지 않는 정비는 허용한다.
- 새로운 규칙이 필요해 보이면 파일을 바로 수정하지 말고 변경 제안만 한다.
- 변경 제안에는 경로, 목적, 적용 범위, 기존 규칙과 충돌 여부, 검증 방법을 포함한다.
