- 주요 사용자는 [사용자 유형 또는 사용 맥락]이다.
- 이 프로젝트에서는 [안정성/속도/감사 가능성/UX/운영 편의성 등]을 우선한다.

## 코드베이스 구조

| 경로 | 역할 | 수정 전 확인할 것 | 위험도/주의사항 |
|---|---|---|---|
| `[경로]` | `[프론트/백엔드/데이터/인프라/공용 라이브러리 등]` | `[관련 문서/테스트/의존 모듈]` | `[건드리면 위험한 이유]` |

- 프론트, 백엔드, 데이터, 인프라, 공용 라이브러리의 경계를 명확히 적는다.

- 특정 영역을 수정하기 전에 반드시 읽어야 하는 문서나 테스트가 있으면 함께 적는다.

## 검증 라우팅

| 변경 영역 | 먼저 확인할 지침 | 비고 |
|---|---|---|
| 프론트엔드 | `[예: apps/web/AGENTS.md]` | UI, 라우팅, 상태, 스타일 변경 |
| API / 백엔드 | `[예: apps/api/AGENTS.md]` | 요청/응답, 인증, 서버 로직 변경 |
| DB / 데이터 | `[예: packages/db/AGENTS.md]` | schema, migration, seed, query 변경 |
| 인프라 / 배포 / CI | `[예: infra/AGENTS.md 또는 docs/deploy.md]` | 배포, 빌드, 워크플로우 변경 |
| 문서 | `[예: docs/AGENTS.md 또는 .agents/workflows/docs-change.md]` | 문서 구조, ADR, 정책 변경 |

- 여러 영역을 건드리면 각 영역의 하위 `AGENTS.md` 검증 규칙을 모두 확인한다.
- 하위 `AGENTS.md`가 없는 영역은 관련 package script, README, 기존 CI 설정을 확인하고 실행 가능한 검증 명령을 사용자에게 보고한다.
- 검증 명령을 찾을 수 없으면 추측해서 만들지 말고, 확인한 후보와 불확실성을 사용자에게 알린다.

## 문서 지도

이 표에는 현재 프로젝트에서 실제 기준으로 사용하는 문서만 적는다. 존재하지 않는 문서나 앞으로 만들 수도 있는 후보 문서는 올리지 않는다.
일반 프로젝트 문서는 이 문서 지도에서 관리하고, agent 지침, workflow, 프로젝트 전용 skill은 `.agents/INDEX.md`에서 관리한다.

| 문서 | 경로 | 상태 | 기준 여부 | 언제 읽는지 | 언제 갱신하는지 |
|---|---|---|---|---|---|
| `[문서명]` | `[경로]` | active | source of truth / 참고 문서 | `[읽는 조건]` | `[갱신 조건]` |

필요한 작업에 해당할 때만 관련 문서를 읽는다.

### 문서 거버넌스 추가 후보

아래 항목은 문서 거버넌스가 필요한 경우에만 검토한다. 기본 경로와 구조는 AOK의 `file-structure.md`를 따른다. 실제로 생성했거나 현재 기준으로 쓰는 문서만 위 문서 지도에 올린다.

| 후보 | 역할 | 문서 지도에 올리는 조건 |
|---|---|---|
| ADR | 제품, 아키텍처, UX, 데이터, 운영 기준 | 현재 의사결정의 source of truth일 때 |
| Current status | 현재 운영 상태, 검증 기준, 배포/외부 의존성, 남은 위험 | 작업 시작 시 확인해야 하는 현재 기준일 때 |
| Version history | 버전, 릴리스 기준, 배포 산출물 | 버전 또는 릴리스 기준을 다루는 프로젝트일 때 |
| Work log | 복구, 롤백, 장애 조사, 회귀 추적용 과거 작업 이력 | 평소 읽는 문서가 아니라 특수 상황에서만 참고할 때 |
| Archive | 폐기, 백업, 원본 참고 자료 | 현재 기준이 아니며 보존 목적일 때 |
| Superpowers outputs | spec, plan, review 결과 | 공식 기준으로 등록된 산출물일 때만 |

`docs/current-status.md`는 현재 믿어야 하는 운영 기준만 짧게 담는다. 과거 변경 이력은 `docs/VERSION_HISTORY.md`, 복구/롤백에 필요한 작업 흐름은 필요한 경우 `docs/work-log.md`, 결정 근거는 ADR에 둔다.

`docs/work-log.md`는 기본 생성 대상이 아니며, 평소 작업 시작 시 읽지 않는다. 장기 운영 메모가 반복되고 기존 issue, PR, plan으로 추적하기 어려울 때 기록 대상으로 만들고, 롤백, 복구, 장애 조사, 회귀 원인 추적처럼 과거 흐름이 필요한 경우에만 읽는다.

ADR이 커졌을 때의 경로와 영역 예시는 AOK의 `file-structure.md`와 `adr.md` 템플릿을 따른다.

## 문서 지도 유지 규칙

프로젝트 문서는 코드처럼 시간이 지나며 낡는다. 문서 지도가 실제 기준 문서와 맞지 않으면 문서 지도를 신뢰하지 않는다.

### 색인 대상

다음 문서만 문서 지도에 올린다.

- 현재 기준으로 사용하는 정책 문서
- 제품, 아키텍처, API, 배포, 데이터, 운영 기준 문서
- 반복적으로 참조되는 작업 가이드
- 최신 의사결정의 source of truth 역할을 하는 문서

다음 문서는 기본적으로 색인하지 않는다.

- 일회성 조사 메모
- 임시 작업 로그
- 오래된 초안
- 백업 파일
- archive에 들어간 참고 자료
- agent가 생성한 중간 산출물 중 공식 기준이 아닌 것

### 갱신해야 하는 경우

다음 변경이 발생하면 같은 작업 안에서 문서 지도도 갱신한다.

- 새 기준 문서를 만들 때
- 기준 문서를 이동, 이름 변경, 삭제할 때
- 어떤 문서가 더 이상 source of truth가 아니게 될 때
- `docs/archive/`로 문서를 이동할 때
- 제품 정책, API 계약, 배포 절차, 데이터 구조, 운영 기준이 바뀔 때
- 문서 간 내용이 충돌해서 하나를 우선 기준으로 정했을 때

### 문서 상태

문서 지도에 올라간 문서는 다음 상태 중 하나를 가진다.

- `active`: 현재 기준 문서
- `draft`: 검토 중인 초안
- `deprecated`: 더 이상 기준으로 쓰지 않지만 전환 참고용으로 남김
- `archived`: 과거 기록 보존용

`deprecated`와 `archived` 문서는 새 작업의 기준으로 사용하지 않는다.

## 작업 절차 문서

- Superpowers를 사용하는 프로젝트에서만 `.agents/workflows/*.md`를 둔다.
- 복잡한 작업은 `.agents/INDEX.md`를 확인하고, 해당 workflow 문서만 읽는다.
- 모든 workflow 문서를 기본으로 읽지 않는다.

## Agent 지침 파일 거버넌스

다음 파일은 관리 대상 정책 파일이다.

- `AGENTS.md`
- `**/AGENTS.md`
- `.agents/**`

이 파일들은 변경 등급에 따라 수정 권한을 구분한다.

### 승인 없이 가능한 변경

다음은 기존 의미를 바꾸지 않는 정비 작업이므로 사용자 승인 없이 수행할 수 있다.

- 오탈자 수정
- 깨진 내부 링크 수정
- 실제 파일 이동에 따른 경로 정정
- 명백히 오래된 날짜나 파일명 정정
- 표 형식 정렬, Markdown 포맷 정리
- INDEX에 이미 승인된 파일의 누락된 메타데이터 보완
- 삭제된 파일을 INDEX에서 `deprecated` 또는 `archived`로 표시하는 정리

단, 변경 후 의미가 달라지지 않아야 한다. 의미가 바뀔 가능성이 있으면 승인 필요 변경으로 다룬다.

### 사용자 승인이 필요한 변경

다음은 사용자 승인 후에만 수행한다.

- 새 `AGENTS.md` 또는 하위 `AGENTS.md` 생성
- `.agents/workflows/*.md` 새로 생성
- 기존 지침의 행동 규칙 변경
- 검증 명령 추가, 제거, 변경
- 적용 범위 변경
- 소유자 또는 책임 경계 변경
- workflow 단계 추가, 제거, 순서 변경
- 문서 생성 정책, 승인 정책, 위험 작업 정책 변경
- agent가 따라야 하는 금지/필수 규칙 변경

### 명시 요청으로 가능한 변경

사용자가 직접 "이 지침을 수정해", "이 workflow를 추가해", "이 구조로 만들어"라고 명시한 경우에는 해당 범위 안에서 수정할 수 있다.

단, 요청 범위를 넘는 추가 지침 생성이나 별도 workflow 추가는 다시 제안 후 승인받는다.

### 애매한 경우

변경이 단순 정비인지 정책 변경인지 애매하면 정책 변경으로 간주하고 먼저 제안한다.

새 하위 `AGENTS.md`, 기존 지침의 의미를 바꾸는 수정, workflow 문서의 의미를 바꾸는 수정이 필요해 보이면 파일을 바로 수정하지 말고 먼저 `.agents/INDEX.md` 또는 AOK의 `agent-change-proposal.md` 양식으로 제안한다.

사용자가 명시적으로 승인한 뒤에만 해당 파일을 생성하거나 의미를 바꾸는 수정을 한다.
승인 후에는 `.agents/INDEX.md`도 함께 갱신한다.

## Placeholder 처리

적용 완료된 `AGENTS.md`에는 `[명령어]`, `[경로]`, `[문서명]`, TODO 같은 미해결 placeholder를 남기지 않는다.

확인하지 못한 값은 본문에 억지로 채우지 않고 아래처럼 분리한다.

```md
## 확인 필요 항목

- `[항목]`: `[왜 확인이 필요한지]`
```
